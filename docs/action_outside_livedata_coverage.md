# livedata 覆盖范围外 action 统计

## 统计口径

本统计用于评估新合并准则下，哪些 `log` action 落在最终训练/输出范围之外。

当前准则：

- 最终训练/输出只保留 `livedata` 有效覆盖片段内的 `110ms` 网格。
- `livedata` 相邻原始点间隔 `> 2000ms` 时切成两个覆盖片段。
- `livedata` 覆盖范围外的 action 不能提前删除，必须先按时间顺序更新隐藏 `state_*`。
- 隐藏 action 更新完成后，再从最终训练/输出表中过滤掉覆盖范围外 rows。

统计方法：

- 解析全部 `sensor/livedata*.txt` 的绝对 timestamp，按 timestamp 的真实日历日期分桶，不依赖文件夹日期。
- 每个真实日历日期内，按绝对时间排序并按 `2000ms` 缺口切分 `livedata` 覆盖片段。
- 解析全部 `log/*log*.txt` 的绝对 timestamp，按 timestamp 的真实日历日期判断 action 是否落在当天任一 `livedata` 覆盖片段内。
- 覆盖外 action 分为：
  - `before`：早于当天第一个 `livedata` 覆盖片段。
  - `after`：晚于当天最后一个 `livedata` 覆盖片段。
  - `between`：落在同一天多个 `livedata` 片段之间的缺口。
  - `no_livedata_day`：当天没有任何 `livedata` 覆盖片段。

## 总体结论

全量 `log` action 共 `570378` 条。

| 类别 | 数量 | 占全部 action |
|---|---:|---:|
| `livedata` 覆盖范围内 | `350449` | `61.44%` |
| `livedata` 覆盖范围外 | `219929` | `38.56%` |

覆盖范围外 action 的原因拆分：

| 原因 | 数量 | 占覆盖外 action |
|---|---:|---:|
| 当天无 `livedata` | `113154` | `51.45%` |
| 在 `livedata` 结束后 | `93431` | `42.48%` |
| 在 `livedata` 开始前 | `12448` | `5.66%` |
| 在多个 `livedata` 片段中间缺口 | `896` | `0.41%` |

主要结论：

- 覆盖范围外 action 不是少量噪声，而是占全部 action 的约 `38.56%`。
- 覆盖外 action 主要集中在两类日期：
  - 当天完全没有 `livedata`，但仍有大量 action。
  - 当天 `livedata` 结束后仍有大量 action。
- 因此，覆盖范围外 action 不能在合并前直接删除；必须先更新 `state_*`，否则后续保留训练行的 IO 状态会不正确。

## 对合并规则的影响

这些统计支持当前合并准则：

1. 最终训练/输出范围可以收缩到 `livedata` 覆盖片段内。
2. 覆盖范围外 action 不能直接丢弃，因为它们数量大，并且会影响后续 `state_*`。
3. 正确流程应该是先处理全量 action，完成 `state_*` 继承，再过滤最终训练行。
4. 对没有 `livedata` 的日期，训练表输出 `0` 行，但 action 仍必须用于更新跨天状态。

### 对 sensor 和 mask 使用策略的影响

结合实际业务含义，当前训练口径固定使用 `150` 个 sensor 列：

- `1..148`：正常真实采集的 sensor 维度。
- `149..150`：默认值补充维度，不作为真实采集 target 或 loss。

因此，处理策略改为：

1. 合并前过滤 raw sensor 行，只接受固定 mask 模板：`mask_1..mask_148 = 1`，`mask_149..mask_150 = 0`。
2. 不满足这个模板的 raw sensor 行视为异常采集行，直接从有效 sensor 候选池中剔除。
3. 当前 `20260416` 的 `34` 个异常 APC 文件中，所有行都不满足固定模板；这些文件应整体从有效 APC 源中排除。
4. 剔除不需要物理删除 raw 文件，只在读取/对齐阶段跳过这些 APC rows/files。
5. 剔除异常 APC 后，如果对应时间仍在 `livedata` 覆盖片段内，优先回退到有效 `livedata_grid`；如果没有有效 `livedata`，不使用异常 APC 生成真实 sensor target。
6. 最终 aligned/cache 仍保留 `150` 个 sensor 数值列；`149..150` 可以保留默认补充值，但 `target_mask` / `loss_mask` 必须排除它们。
7. `mask_*` 在最终数据中可以按固定模板生成用于兼容旧流程；是否把 `mask_*` 喂给模型作为 input feature，可以后续单独做 ablation。
