# 项目背景

## 1. 项目目标

这个项目的目标，是把半导体设备的三类数据对齐到 `livedata` 覆盖范围内的固定 `110ms` 时间网格上，生成可输入神经网络的逐日训练表，并支持后续 GRU 模型进行长期闭环预测：

- `log`：控制命令 / action
- `apc`：高频分段 sensor 采集
- `livedata`：较低频 sensor 采集，当前数据通常只覆盖白天约 `12` 小时

最终希望每一个 `110ms` 网格行同时表达三件事：

1. 这个时刻设备的 sensor 值是什么
2. 这个时刻哪些 `io_id` 发生了 action
3. 这个时刻所有 `io_id` 当前分别处于什么状态

项目最终目标不是只做单步拟合，而是让模型在没有任何真实 sensor 数据修正的情况下，使用自身历史预测结果继续向后滚动，稳定闭环预测一个月。

当前项目训练/输出只保留有 `livedata` 覆盖的时间段。夜间或其他 `livedata` 覆盖范围外的网格不进入最终训练表；但这些区间里的 action 不能先删除，必须先按时间顺序更新 `state_*`，再过滤输出行。这样后续 `livedata` 覆盖窗口开始时，`state_*` 已经继承了窗口外 action 的影响。

`20260318` 是当前数据集的开机首日，首条 `livedata` 为 `2026-03-18 11:14:23.321`，首条 action 为 `2026-03-18 11:14:44.035`。当天 `00:00:00.000` 到首条 `livedata` 之前只作为初始化阶段：`state_*` 使用 `action_default.xlsx` 初始化，如有 action 则隐藏更新；不输出 pre-`livedata` 训练行。

## 2. 输入数据含义

### `log`

`log` 表示控制命令，只在 action 发生时记录一行，核心字段是：

- `timestamp`：action 发生时间
- `io_id`：阀门或 IO 通道
- `io_value`：这次 action 后该通道的状态值

### `apc`

`apc` 是高频、分时间段采集的 sensor 数据，采样频率约 `110ms`。

特点：

- 文件头有 `Process Start Time`
- 表内 `Time` 不是绝对时间，而是相对于起点的偏移值
- 使用前需要先恢复成绝对时间

### `livedata`

`livedata` 是较低频 sensor 数据，采样频率大约在 `300ms ~ 1000ms`，当前数据通常只覆盖白天采集窗口。

特点：

- 时间本身就是绝对时间
- 密度低于 `apc`
- 在没有 `apc` 覆盖的区域，用来补足有真实采集支撑的 sensor 时间序列
- 没有 `livedata` 的夜间不等价于设备关闭；最终训练表不输出这些区间，但其中的 action 仍要更新后续 `state_*`

## 3. 输出目标

最终每天生成两张表：

- 原始对齐表：便于核对和追溯
- 训练表：便于直接输入模型

输出表会在原始 sensor 列之外，再展开所有 `io_id` 的：

- `evt_{id}`：该时刻是否发生 action，取值 `0/1`
- `state_{id}`：该时刻该 IO 的持续状态

训练与评估时，应把这些输出视为长期时序建模的基础数据。`sensor` 表达被预测对象，`evt_*` 表达该网格内发生过的控制动作，`state_*` 表达 IO 的持续状态上下文。
